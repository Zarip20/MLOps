# Постановка задачи и проектирование системы

## 1. Источник данных

- **Набор данных:** Ethiopian Insurance Corporation (страхование транспортных средств), 2014–2018 гг.
- **Формат:** CSV, ~500 000 записей, 16 колонок.
- **Временная переменная:** `INSR_BEGIN` (дата начала полиса) — используется для разбиения на батчи (по месяцам).
- **Целевая переменная:** `HAS_CLAIM` — факт наличия страхового случая (1, если `CLAIM_PAID` не пусто, иначе 0). Классы сильно несбалансированы.

## 2. Прогнозируемая задача

**Бинарная классификация:** определить, относится ли клиент к высокорисковой группе (вероятность страхового случая).

## 3. Этапы обработки данных

### 3.1. Сбор и эмуляция потока
- Исходный CSV разбивается на месячные батчи с помощью `split_into_batches()`.
- Каждый батч сохраняется в `data/raw_batches/batch_YYYY-MM.csv`.
- Состояние обработки хранится в `state.json` (список батчей, индекс последнего обработанного).

### 3.2. Оценка качества данных (Data Quality)
Для каждого батча вычисляются:
- **Completeness:** общая доля пропусков, максимальная доля пропусков в столбце и строке.
- **Validity:** для числовых признаков проверяется неотрицательность.
- **Timeliness:** минимальная/максимальная дата, максимальный разрыв между датами.
- **Статические правила:** 5 правил из `config.yaml` (например, `PREMIUM >= 0`). Доля нарушений сохраняется.
- **Динамические правила (Apriori):** на первом батче генерируются 5 ассоциативных правил с максимальной уверенностью. В следующих батчах вычисляется доля строк, нарушающих эти правила (например, `MAKE = AFRO → INSR_TYPE = 1202`).

### 3.3. Очистка данных
- Удаляются строки, где доля пропусков > 50%.
- Удаляются строки, нарушающие статические правила, если доля таких нарушений в батче ≤ 10%.

### 3.4. Предобработка (Preprocessing)
Используется `ColumnTransformer` с фиксированными списками колонок из `config.yaml`:
- **Числовые признаки:** заполнение пропусков медианой, затем `StandardScaler`.
- **Категориальные признаки:** заполнение пропусков наиболее частым значением, затем `OneHotEncoder` с `handle_unknown='ignore'`.

Препроцессор обучается на первом батче и сохраняется, для последующих батчей он только применяется (`transform`).

## 4. Модели и гиперпараметры

### 4.1. Дерево решений (`DecisionTreeClassifier`)
- **Гиперпараметры (из config.yaml):** `max_depth=10`, `min_samples_split=20`, `min_samples_leaf=10`.
- **Обучение:** переобучается с нуля на всех накопленных данных (каждый `update`).

### 4.2. Нейронная сеть (`MLPClassifier`)
- **Гиперпараметры (из config.yaml):** `hidden_layer_sizes=[50,25]`, `activation='relu'`, `solver='adam'`, `max_iter=200`, `warm_start=True`.
- **Дообучение:** при наличии сохранённой модели вызывается `partial_fit` (инкрементальное обучение). Иначе — обучение с нуля.

## 5. Валидация и метрики

- **Схема:** hold-out (20% текущего батча откладывается для валидации).
- **Метрики для каждой модели:** precision, recall, f1-score, ROC-AUC.
- **Выбор лучшей модели:** по **f1-score** (устойчив к дисбалансу классов).
- **Хранилище:** каждая модель сохраняется как `*_latest.pkl`, лучшая — `best_model.pkl`. Метрики сохраняются в `data/metadata/model_metrics_<batch_idx>.json`.

## 6. Сохранение артефактов

| Артефакт | Путь | Назначение |
|----------|------|-------------|
| Батчи исходных данных | `data/raw_batches/` | Эмуляция потока |
| Метаданные качества | `data/metadata/quality_*.json` | История качества по батчам |
| Метаданные метрик моделей | `data/metadata/model_metrics_*.json` | История метрик |
| Ассоциативные правила | `data/rules.json` | Проверка корректности |
| Препроцессор | `models/preprocessor.pkl` | Единая предобработка |
| Лучшая модель | `models/best_model.pkl` | Инференс |
| Модели для дообучения | `models/mlp_latest.pkl`, `models/dt_latest.pkl` | Продолжение обучения |
| Отчёты | `reports/summary_*.txt` | Мониторинг |

## 7. Интерфейс командной строки

- `python run.py -mode init` — разбиение на батчи.
- `python run.py -mode update` — обработка следующего батча (качество, обучение/дообучение, сохранение).
- `python run.py -mode inference -file <путь>` — предсказание на новых данных.
- `python run.py -mode summary` — формирование текстового отчёта.

## 8. Конфигурация (config.yaml)

Все ключевые параметры вынесены в `config.yaml`:
- пути к папкам;
- список признаков, категориальных и числовых колонок;
- статические правила качества;
- гиперпараметры моделей.

## 9. Логирование

Используется модуль `logging` (уровень INFO). В консоль выводятся сообщения о загрузке батчей, создании препроцессора, размерности признаков, сохранении моделей.

## 10. CI/CD с GitHub Actions

Проект оснащён workflow **MLOps CI Pipeline**, который автоматически:
- устанавливает Python и зависимости,
- разбивает данные на батчи,
- последовательно обрабатывает все N батчей (дообучает модели),
- генерирует итоговый отчёт summary,
- сохраняет логи и отчёты как артефакты.

Workflow запускается при каждом push или pull request в ветку `main`, а также может быть запущен вручную через интерфейс GitHub Actions.

Был выбран итеративный подход, потому что базовый слишком скучный, а для обработки любого количества батчей проще реализовать последовательный итеративный процесс, вместо создания/загрузки расписания и сохранения модели и состояния для каждого запуска workflow/

### Получение артефактов

1. Вкладка **Actions** репозитория.
2. Выбор workflow **MLOps CI Pipeline** и запуск через `Run workflow` (благодаря workflow_dispatch:).
3. Внизу страницы будет раздел **Artifacts** с архивом `training-artifacts`, внутри будут `training.log` и текстовый отчёт.
